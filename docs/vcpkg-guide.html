vcpkg 使用大全

目标：把 vcpkg 当「C++ 包管理器」用熟，特别是 CMake 项目 场景。
本文以 Windows + Visual Studio 2022 为主，Linux/macOS 差异会单独标注。

1. 什么是 vcpkg

vcpkg 是 Microsoft 主导的 C/C++ 包管理器，特点：

  • 包存储在本地仓库（默认 C:\vcpkg），不污染系统
  • 自动下载、编译、安装第三方库
  • 与 CMake / MSBuild / Visual Studio 三种集成方式
  • 跨平台：Windows / Linux / macOS 都跑

常见替代品：Conan、xmake、system package manager（apt / brew）。

2. 安装与初始化

2.1 Windows 安装

# 1. 选个目录，比如 C:\vcpkg
git clone https://github.com/microsoft/vcpkg.git C:\vcpkg
cd C:\vcpkg
.\bootstrap-vcpkg.bat

完成后会生成 vcpkg.exe。

2.2 环境变量

vcpkg 不强制要求环境变量，但 CMake 集成时需要 $env{VCPKG_ROOT} 指向 vcpkg 根目录（很多 CMakePresets.json 用这个变量）：

[System.Environment]::SetEnvironmentVariable('VCPKG_ROOT', 'C:\vcpkg', 'User')

新开终端生效。

3. 基本命令

命令用途
vcpkg search <pkg>搜包（也支持模糊匹配，比如 vcpkg search glfw）
vcpkg install <pkg>装包到 default triplet（Windows 默认 x64-windows）
vcpkg install <pkg>:<triplet>指定 triplet，比如 x64-windows-static
vcpkg list列出已装包
vcpkg remove <pkg>卸载
vcpkg update拉取 ports 最新版本（不重装已有包）
vcpkg upgrade升级所有过期包
vcpkg hash <file>给 ports 文件做 hash（改 portfile 后会用）
vcpkg x-update-baseline生成 manifest mode 的 baseline 版本

示例：

vcpkg search glm
# glm                    1.0.1#1     ...
# glm                    0.9.9+1#2  ...

vcpkg install glfw3 glm stb tinyobjloader

4. Triplet 详解

Triplet 是「目标平台 + 链接方式」的组合，决定库装成动态还是静态、用哪个 CRT。

4.1 常用 triplet

Triplet含义
x64-windowsWindows x64，动态链接 CRT (/MD)，动态库 (.dll)，默认
x64-windows-staticWindows x64，静态 CRT (/MT)，静态库 (.lib)
x64-windows-release只装 Release 配置
x64-windows-dynamic等价于 x64-windows
x64-linux / x64-osx跨平台

4.2 选哪个

  • 普通应用想跑就得动态链接（x64-windows），依赖 vcpkg 部署 dll
  • 想生成一个「给别人就能用、不带 vcpkg」的 exe？选 x64-windows-static
  • 静态链接导致不同库的 CRT 不一致经常炸，所以项目里只用一个 triplet

5. 与 CMake 集成

三种方式，按推荐度排序：

5.1 通过 CMakePresets.json（最推荐）

CMakePresets.json：

{
  "version": 6,
  "configurePresets": [
    {
      "name": "vcpkg-debug",
      "generator": "Visual Studio 17 2022",
      "architecture": "x64",
      "toolchainFile": "$env{VCPKG_ROOT}/scripts/buildsystems/vcpkg.cmake",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "Debug"
      }
    }
  ]
}

VS 2022 打开项目会自动列出这个 preset。命令行：

cmake --preset vcpkg-debug
cmake --build --preset debug

5.2 命令行直接指定 toolchain

cmake -S . -B build -G "Visual Studio 17 2022" -A x64 ^
  -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake

5.3 在 CMakeLists.txt 里硬写（不推荐）

set(CMAKE_TOOLCHAIN_FILE "C:/vcpkg/scripts/buildsystems/vcpkg.cmake"
    CACHE STRING "")
这种写法把 toolchain 路径写死在源码里，不可移植。一定用 5.1 或 5.2。

5.4 vcpkg toolchain 做了什么

加载 vcpkg.cmake 后：

  • 设置 VCPKG_INSTALLED_DIR、VCPKG_TARGET_TRIPLET（Vulkan 项目里这两个最常用）
  • 重写 find_package：包存在 vcpkg 时优先用 vcpkg 版本，自动 include vcpkg-cmake-wrapper.cmake
  • 配合 MSBuild 时调 applocal.ps1 把 dll 复制到 exe 旁边

6. Manifest mode（强烈推荐）

项目里加一份 vcpkg.json 声明依赖。新人 clone 项目后 vcpkg install 自动装齐所有。

6.1 启用方式

CMake 里只要 toolchain 文件加载了，看到当前目录有 vcpkg.json 就自动进 manifest mode，不需要额外 enable。

6.2 vcpkg.json 示例

{
  "name": "vulkan-tutorial",
  "version-string": "0.1.0",
  "dependencies": [
    "vulkan",
    "glfw3",
    "glm",
    "stb",
    "tinyobjloader"
  ],
  "builtin-baseline": "abc123..."
}
builtin-baseline 锁定到某个 commit —— 没这个字段默认拿 ports 当前 HEAD。 运行 vcpkg x-update-baseline --add-initial-baseline 自动生成。

6.3 vcpkg-configuration.json（可选）

中央版本覆盖、registries 配置：

{
  "default-registry": {
    "kind": "git",
    "repository": "https://github.com/microsoft/vcpkg",
    "baseline": "abc123..."
  },
  "registries": [
    {
      "kind": "git",
      "repository": "https://my-internal-mirror/vcpkg-registry",
      "baseline": "def456...",
      "packages": ["internal-lib"]
    }
  ]
}

7. 怎么查一个包的 target 名（重点）

新手最大痛点：find_package(<pkg>) 之后该链接 <pkg> / <pkg>::<pkg>？ 别猜，去查。

7.1 安装路径

vcpkg 装好的包在 <vcpkg>/installed/<triplet>/share/<pkg>/。

Windows 默认路径：

C:\vcpkg\installed\x64-windows\share\glm\
├── copyright
├── glmConfig-debug.cmake
├── glmConfig-release.cmake
├── glmConfig.cmake          ← 这个
├── glmConfigVersion.cmake
├── usage                     ← 这个
└── vcpkg-cmake-wrapper.cmake

7.2 看 usage 文件（最快）

Get-Content C:\vcpkg\installed\x64-windows\share\glm\usage

输出：

The package glm provides CMake targets:

    find_package(glm CONFIG REQUIRED)
    target_link_libraries(main PRIVATE glm::glm)

直接抄。

7.3 没 usage 就 grep targets 文件

Select-String -Path "C:\vcpkg\installed\x64-windows\share\glfw3\glfw3Targets.cmake" -Pattern "add_library"

输出类似：

add_library(glfw SHARED IMPORTED)

这就是 target 名（glfw，不是 glfw::glfw —— 历史命名遗留）。

7.4 看 Portfile.cmake（最权威）

去 vcpkg 仓库 ports/<pkg>/portfile.cmake，搜 vcpkg_install_cmake 后面跟的 TARGETS 或 EXPORT。

# 克隆了 vcpkg 源码后可看：
Get-Content C:\vcpkg\ports\glm\portfile.cmake

7.5 一句话口诀

先 usage，没就 grep add_library，再不行翻 portfile.cmake。

8. find_package 模式的选择

vcpkg 的包提供两种「包描述文件」之一：

模式文件名示例谁提供
ModuleFind<X>.cmakeFindStb.cmakevcpkg / 系统 / 用户
CONFIG<x>-config.cmake 或 <X>Config.cmakeglmConfig.cmake、tinyobjloader-config.cmake上游项目原生

8.1 find_package 怎么选

# 默认（先 module 再 config）
find_package(<pkg> REQUIRED)

# 强制 CONFIG（vcpkg 主流包都用这种，推荐）
find_package(<pkg> CONFIG REQUIRED)

# 强制 Module（少见）
find_package(<pkg> MODULE REQUIRED)

8.2 vcpkg 的「混搭」规则

vcpkg 自动加 vcpkg-cmake-wrapper.cmake 把 Find<X>.cmake 塞到 CMAKE_MODULE_PATH，所以不带 CONFIG 也能找到 Find 模块（比如 stb）。

但如果上游只提供 config 文件（比如 glm），就必须 CONFIG。

判断方法：

装了哪些文件写法
只有 Find<X>.cmake，没 configfind_package(<pkg> REQUIRED)
只有 <pkg>Config.cmakefind_package(<pkg> CONFIG REQUIRED)
两者都有任意

9. 实用片段

9.1 自动部署 dll

vcpkg + MSBuild 已经会调 applocal.ps1 把依赖 dll 复制到 exe 旁。 Debug / Release 分别用 <vcpkg>/installed/<triplet>/debug/bin 和 /bin。

如果你想 CMake 自己处理（确保 applocal.ps1 没救），在 CMakeLists.txt 里加：

if(WIN32)
    add_custom_command(TARGET MyApp POST_BUILD
        COMMAND ${CMAKE_COMMAND} -E copy_if_different
            "${VCPKG_INSTALLED_DIR}/${VCPKG_TARGET_TRIPLET}/$<CONFIG>/bin/<dllname>.dll"
            $<TARGET_FILE_DIR:MyApp>
    )
endif()

$<CONFIG> 是 generator expression，VS 多配置 generator 下会自动替换成 Debug 或 Release。

9.2 同时装多个 triplet

x64-windows 是默认；想编译两份（比如一个静态一个动态）：

vcpkg install glfw3:x64-windows glfw3:x64-windows-static

装两份之后 <vcpkg>/installed/ 下会有两个 triplet 目录：

C:\vcpkg\installed\x64-windows\
C:\vcpkg\installed\x64-windows-static\

切换 triplet：清 build 目录 + 重新 configure + 重新指 toolchain。

10. 故障排查

10.1 「找不到 XConfig.cmake / X-config.cmake」

  1. vcpkg toolchain 是否加载？检查 CMakeCache.txt 里 CMAKE_TOOLCHAIN_FILE 值
  2. 包是否装了？vcpkg list
  3. 包是否提供 config 文件？查 installed/<triplet>/share/<pkg>/ 目录
  4. 删 build 目录重配：Remove-Item -Recurse build

10.2 「Target X not found」

  1. vcpkg 装了 ≠ CMake 看到了 → 看是否真的用 toolchain 配的
  2. target 名写错了 → 查 usage 或 grep add_library
  3. alias 缺失 → 改用包提供的实际 target 名

10.3 applocal.ps1 失败

applocal.ps1 需要 dumpbin（或 llvm-objdump）扫描 exe import 表。VS 完整装好后 PATH 里有 dumpbin。 如果单独跑 cmake 时找不到，VS Developer Command Prompt 里跑，或者把 vcpkg 那段 MSBuild target 禁掉，自己手复制 dll。

10.4 静态 vs 动态 CRT 冲突

报错类似：

LNK2038: mismatch detected for 'RuntimeLibrary': value 'MT_StaticRelease' doesn't match 'MD_DynamicRelease'

原因：项目 /MT，依赖 /MD（或反过来）。

解法：换 triplet 让两边一致；或者显式在 target_compile_options / target_link_options 里覆盖。

10.5 Vulkan 相关

  • vcpkg 的 vulkan 包只是个 wrapper（不装 runtime dll）
  • vulkan-loader 提供 vulkan-1.dll，debug/release 都有
  • 显卡驱动通常也装一份 vulkan-1.dll 到 C:\Windows\System32\
  • Vulkan SDK 1.3+ 默认不包含 runtime dll（Bin 目录里没有）
