1. 总体定位

当前代码不是把物体、相机、光源和 GPU 命令混在一个大对象中，而是用 RHIFramePacket 作为一帧的交换边界，再由 RenderGraph 把“想得到什么结果”编译成后端可以线性执行的计划。

一帧从引擎到屏幕的主路径

Game / ECSEntity、Transform、Camera、Light
→
Render Collector裁剪、分类、排序、收集
→
RHIFramePacket分离的输入与命令声明
→
RHIDevice哈希、缓存、分派后端
→
ExecutionPlan排序、裁剪、槽与 barrier
→
GPU / PresentVulkan、D3D11、D3D12
最重要的边界 RHIRenderGraphDesc 描述依赖，RHIRenderPassWorkload 描述实际命令，RHIRenderGraphExecutionPlan 是编译结果。图声明不应直接调用 Vulkan 或 Direct3D。

上层负责

从 ECS/场景中收集可见对象、相机和光源，选择渲染路径，创建 pass、资源引用与 draw 数据。

RenderGraph 负责

校验声明、推导 RAW/WAR/WAW、稳定拓扑排序、裁剪无效 pass、计算生命周期和同步需求。

RHI 后端负责

把抽象资源、状态与命令翻译成原生 API，对象生命周期必须覆盖 GPU 实际使用时间。

2. 分层结构

这套结构用数据和组合连接各层，不要求 Vulkan、D3D11、D3D12 继承一个庞大的设备基类。RHIDevice 通过 std::variant 保存一个具体实现并统一转发。

层当前代码职责不应该做的事
公共定义RHIDefinitions.hpp句柄、枚举、描述体、FramePacket、Graph 声明包含任一图形 API 原生类型
设备门面Core/RHIDevice.*选择实现、统一资源 API、缓存编译计划重复实现具体资源创建
图编译器RenderGraph/RHIRenderGraph.*把声明编译成纯索引执行计划创建 VkImage 或 ID3D12Resource
后端Vulkan/、D3D11/、D3D12/资源、管线、同步、命令与提交重新决定 pass 依赖顺序
示例PBRDemo/main.cpp构建真实 FramePacket 并提交直接越过 RHI 调用图形 API
验证Tests/检查编译算法和三后端执行只检查“程序没有崩溃”

为何计划和当前帧数据要分开

拓扑、资源描述、pass 类型等通常连续多帧相同；clear color、viewport、draw 数量、upload 字节和 swapchain image 句柄却会变化。缓存整个 packet 会读到上一帧数据，完全不缓存又会每帧重复编译。

静态结构 ──Hash──> RHIRenderGraphExecutionPlan（可缓存）
当前帧数据 ──────> RHIFramePacket（永远使用本帧对象）

后端执行 = 缓存 plan 中的整数下标 + 当前 packet 中的动态值

4. 完整帧流程

从 PBRDemo::DrawFrame 到 GPU 完成，一帧依次经过以下路径。这里的顺序也适合作为调试时的断点顺序。

运行时调用链

1

AcquireNextImage

取得 swapchain image 下标，并建立窗口系统到图形提交之间的 GPU wait 信号。

2

BuildFrame

填充 settings、uploads、导入资源、pass、workload、submission 和 present。

3

RHIDevice::SubmitFrame

计算结构哈希。缓存未命中时调用 CompileRHIRenderGraph，命中时直接复用计划。

4

Validate submissions

确认 passNames 没有遗漏、重复或违反 compiled dependency 顺序。

5

解析图资源

Imported 资源绑定当前外部句柄；内部 transient 资源按 physical slot 创建。

6

执行 uploads

CPU 可见资源直接写；GPU-only 资源经 staging 和 copy，并更新后端真实状态。

7

线性执行 compiled passes

每个 pass 先应用 transitions，再执行 copy、clear、draw 或 dispatch。

8

提交与 Present

处理外部 wait/signal，提交命令流，转为 Present 状态并交给窗口系统。

9

延迟回收

使用 timeline/fence 判断 GPU 完成，之后才能复用帧上下文或释放临时 native 资源。

CPU 调用顺序不等于 GPU 完成顺序 提交函数返回通常只表示命令已经进入队列。staging buffer、descriptor、command allocator 和 transient resource 不能因此立刻销毁或覆盖，必须等待对应 timeline/fence 值完成。

5. RenderGraph 编译十阶段（0–9）

CompileRHIRenderGraph 把容易编写、包含字符串的声明转换成后端友好的整数计划。编译不是录制 GPU 命令，它只做 CPU 侧分析。

阶段输入核心操作输出/错误
0资源/pass 数量初始化 lifetime、slot、culled 数组并计算结构哈希固定大小结果容器
1资源/pass/workload 名称建立 name → index 表，校验唯一性与 imported handle整数解析表
2pass + workload校验命令类别；合并 reads、writes、attachments 为规范化 usagePassBuildData
3dependsOnPasses只对显式依赖做第一次稳定拓扑排序hazard 扫描基准顺序
4规范化 usages跟踪 lastWriter/readers，补 RAW/WAR/WAW 边完整数据依赖图
5完整依赖图第二次稳定拓扑排序并检测环最终 source pass 顺序
6根 pass 与 dependencies从副作用/Present/imported/output 反向标记 live pass裁剪结果
7live source passes转换为紧凑 compiled index，解析 attachment/workload 下标线性 pass 计划
8compiled usages计算 first/last pass；贪心分配兼容 physical slot生命周期与槽映射
9槽前任 + 逻辑状态补 allocation dependency，生成 state/stage/access/queue transitions最终 ExecutionPlan

为什么要排序两次

用户可能把 consumer 写在 producer 前面，再用 dependsOnPasses 指定真实顺序。如果直接按声明顺序扫描资源，编译器会错误地认为 consumer 先读了未初始化资源。第一次排序只建立可信扫描顺序，扫描后加入 hazard 边，再进行第二次最终排序。

稳定拓扑排序

算法使用 indegree 和 ready 队列。多个 pass 同时 indegree 为 0 时，优先选择原始下标更小的 pass。这样不相关 pass 的顺序可重复，GPU capture 和测试不会随机变化。

建立 dependency -> dependent 邻接表
计算每个 pass 的 indegree
把 indegree == 0 的 pass 按原始下标加入 ready

while ready 非空:
    取最小原始下标 pass
    输出到 order
    对每个 dependent: --indegree
    indegree 变为 0 时按下标插入 ready

order.size() != passCount  => 图中存在环

6. 数据依赖、环与裁剪

依赖不是为了让代码“看起来有顺序”，而是保护同一资源上的读写关系。编译器为每个资源维护一个 lastWriter 和一组尚未被下一次写入截断的 readers。

三类 hazard 如何产生边

RAW
Pass A: Write X
→
Pass B: Read X
WAR
Pass A/B: Read X
→
Pass C: Write X
WAW
Pass A: Write X
→
Pass B: Write X

RAW

后一次读取必须看到前一次写入。reader 依赖 lastWriter，这是最常见的 producer → consumer。

WAR

后一次写入不能提前覆盖仍在读取的数据。writer 依赖当前累计的全部 readers，然后清空 reader 集合。

WAW

两个写入必须保持定义顺序，否则最终结果不确定。新 writer 依赖 lastWriter。

未初始化读取

Imported 资源被认为已经在图外初始化；内部 texture 若有非 Undefined initialState 也可作为已初始化。其余内部资源首次访问必须写，否则编译返回错误。

Pass 裁剪

编译器先寻找根 pass，再沿 dependencies 反向标记。根包括：

  • cullable == false 或 hasSideEffect == true；
  • Present pass；
  • 写入 imported、exported 或 NeverCull 资源的 pass。

未被任何根依赖的 pass 会被裁剪。例如一个调试 compute pass 写了内部临时纹理，但没有后续 pass 读取，也没有副作用，它不需要提交给 GPU。

声明副作用 GPU readback、统计计数、外部共享资源写入等结果可能不在图内被读取。此类 pass 必须设 hasSideEffect，或把资源标记为 exported/NeverCull，否则裁剪是合理行为。

7. 生命周期与 transient 别名

每个逻辑资源的生命周期是它第一次和最后一次出现在 compiled pass 中的位置。只有 previous.lastPass < next.firstPass 才允许共享物理槽；等号表示两个资源在同一 pass 仍同时存活。

8. Barrier 为什么以及怎样生成

依赖边回答“谁必须先于谁”；barrier 回答“前面的写入何时结束、何时对后面的访问可见、资源应处于什么状态”。只有顺序没有内存可见性，在显式 API 中仍可能读到错误数据。

字段回答的问题VulkanD3D12
before / after逻辑用途如何变化？映射 image layout 与默认 stage/access映射 D3D12_RESOURCE_STATES
sourceStages要等待哪些生产阶段？srcStageMask由 state barrier 模型隐含
destinationStages哪些消费阶段可继续？dstStageMask由 state barrier 模型隐含
sourceAccess哪些旧访问要完成/flush？srcAccessMask由 state/UAV barrier 表达
destinationAccess哪些新访问需要可见？dstAccessMask由 state/UAV barrier 表达
sourceQueue / destinationQueue是否跨队列？未来扩展 ownership transfer未来扩展 queue fence

何时生成 transition

出现以下任一条件时，编译器为 pass 添加 transition：

  • 资源尚未初始化；
  • 逻辑状态发生变化；
  • 目标队列发生变化；
  • 上一次访问写入，或本次访问要写入。

如果是纯 read → read，状态和队列都相同，则可省略 barrier。

discardContents 与 aliasingBarrier

discardContents

当前 pass 不关心旧内容，例如 attachment 的 loadOp 是 Clear/DontCare。Vulkan 可以从 UNDEFINED layout 开始，避免保留像素内容。

aliasingBarrier

物理槽刚从另一个逻辑资源切换过来。即使新资源要 Clear，也必须先等待前任资源最后一次访问结束，才能安全复用同一 native 对象。

discard = true, aliasing = false
    => 旧内容与旧访问都可忽略，Vulkan 可使用 TOP_OF_PIPE + UNDEFINED

discard = true, aliasing = true
    => 内容可以不要，但前任逻辑资源的 GPU 访问不能忽略
    => 必须保留真实 old stage/access/layout 作为同步起点

为何编译器状态和后端状态同时存在

编译器只知道图内逻辑用途。Imported buffer 可能刚被 staging copy，swapchain image 可能刚 Acquire，资源也可能在图外使用过。后端追踪的 native 当前状态才是 barrier 的真实起点；编译器 transition 提供目标和依赖意图。

9. Vulkan、D3D11、D3D12 如何执行同一计划

三个后端消费相同 RHIRenderGraphExecutionPlan，但原生 API 的状态与提交模型不同。统一的是语义，不是强行让原生代码长得一样。

主题VulkanD3D11D3D12
命令模型录制 VkCommandBuffer直接调用 immediate context录制 graphics command list
状态同步显式 stage/access/layout barrier驱动隐式管理；RHI 只追踪逻辑状态显式 transition/UAV barrier
帧完成内部 timeline semaphore 递增值immediate context/驱动持有引用fence value + event
临时上传host-visible staging bufferMap/UpdateSubresourceUPLOAD heap staging resource
Descriptordescriptor set绑定到 shader slots复制到 shader-visible heap
Attachmentdynamic renderingRTV/DSV + OMSetRenderTargetsRTV/DSV descriptor + OMSetRenderTargets

Vulkan 执行重点

  • 上传 copy 后立即发 buffer memory barrier，并更新真实 currentStages/currentAccess。
  • 每个 compiled pass 前，把 transition 转成 VkBufferMemoryBarrier 或 VkImageMemoryBarrier。
  • image barrier 同时处理 layout、stage 和 access；alias 时不能错误地把 oldLayout 当成 UNDEFINED。
  • 每次图形提交在内部 frame timeline semaphore 上 signal 严格递增值。FrameContext 保存自己的 completionValue。
  • 下次复用 FrameContext 前等待 timeline 到达对应值，再释放 staging 和延迟销毁资源。

D3D11 执行重点

  • RenderGraph 依然负责依赖、排序和裁剪，但 D3D11 没有 Vulkan/D3D12 式 resource state barrier。
  • compiled transition 更新 RHI 逻辑状态，原生资源冲突由运行时/驱动处理。
  • workload 通过 RHIBindSet 进入统一绑定逻辑，再铺到 VS/PS/CS slot。
  • pass 顺序直接体现为 immediate context 方法调用顺序。

D3D12 执行重点

  • 单 allocator 版本在 Reset 前等待上一 command list 的 fence value 完成。
  • 普通状态变化发 TRANSITION barrier；UAV 状态不变但连续读写时发 UAV barrier。
  • UPLOAD/READBACK heap 的状态受 D3D12 固定约束，不按 DEFAULT heap 方式 transition。
  • bind 时把持久 CPU descriptor 复制到帧内 shader-visible heap，heap 复用受 fence 保护。
  • staging 和 transient 所有权在提交后移入 pending*，下一次确认 fence 完成后释放。

编译结果如何形成

  1. ShadowMap 写 ShadowDepth；OpaquePBR 读同一纹理并要求 ShaderRead，因此产生 ShadowMap → OpaquePBR 的 RAW 边和 DepthWrite → ShaderRead transition。
  2. OpaquePBR 写 BackBuffer attachment；Present 读同一资源并要求 Present 状态，因此产生第二条 RAW 边。
  3. OpaquePBR 写入 imported BackBuffer/Depth，Present 本身也是根；反向遍历会保留三个 Pass。
  4. BackBuffer 从后端真实 Present 状态转换到 RenderTarget，再在帧尾转回 Present。
  5. ShadowMap workload 只提交球体的 depth-only indexed draw；OpaquePBR workload 依次绘制球体、地面和天空盒。
  6. 当前示例所有 graph 资源都是 imported，因此不会展示 transient physical slot alias；相关行为由单元测试覆盖。

11. 新增一条渲染路径时怎样组织

添加 Shadow、GBuffer、Lighting、PostProcess 或 UI 时，上层只应声明资源流和 workload。不要在业务层手工猜测 barrier，也不要让后端重新推导 pass 顺序。

典型延迟渲染路径

UploadScene buffers
→
ShadowWrite ShadowDepth
→
GBufferWrite MRT + Depth
→
LightingRead GBuffer / Write HDR
→
PostProcessRead HDR / Write BackBuffer
→
PresentRead BackBuffer

推荐步骤

  1. 先列出 pass 的输入输出，不写 API 代码。
  2. 明确每个资源是 imported、transient 还是 exported；只有真正帧内临时资源才允许 alias。
  3. 为每次访问填写准确 state、stage、access。能由 state 推导时 access 可保持 None。
  4. 用资源读写表达依赖；只有无资源关系的顺序才使用 dependsOnPasses。
  5. 把实际 draw/dispatch/copy 放进同名 workload。
  6. 先写编译器测试，检查 pass 顺序、裁剪、lifetime、slot 和 transition。
  7. 再逐后端实现命令翻译；未实现能力必须返回清晰错误。

  • 只看 graph 声明，能否画出资源 producer/consumer 图？

12. 当前边界、验证与下一步

文档描述的是当前已经存在的能力，也要明确尚未完成的部分。显式失败比静默跳过更适合作为可持续扩展的基础。

已经具备

  • 显式依赖与 RAW/WAR/WAW 推导
  • 稳定拓扑排序和环检测
  • pass 裁剪与资源生命周期
  • 兼容 transient 物理槽复用
  • alias dependency 与 barrier 标记
  • 结构哈希与执行计划缓存
  • 三后端 buffer copy 和 PBRDemo 路径
  • Vulkan timeline 帧完成追踪

当前限制

  • 每后端当前以一个主 command stream/list/context 执行一帧
  • queue 字段尚未形成真正异步 compute / transfer 多队列调度
  • 高级 texture copy/blit/mipmap workload 尚未完整实现
  • query 与 indirect draw/dispatch 尚未实现
  • workload 内显式 barrier 被禁止，应声明 graph reads/writes
  • transition 主要按整个资源处理，未做细粒度 subresource 跟踪
  • transient 复用尚未升级为跨帧资源池/原生 heap alias

现有测试覆盖

目标覆盖内容对应目标
RHIRenderGraphTestshazard、裁剪、生命周期、环、校验、哈希、alias slot/barrier、submission 顺序纯 CPU 编译算法
RHIBackendGraphTests可用后端执行 attachment frame 与 buffer copy后端计划消费路径
PBRDemoswapchain、upload、indexed sphere/plane、PBR shader、present端到端可视结果

建议的后续优化顺序

  1. 先实现跨帧 transient resource pool，减少每帧 create/destroy。
  2. 把 barrier 跟踪细化到 mip/layer 或 buffer range。
  3. 将 compiled pass 分组为 queue batches，生成跨队列 wait/signal。
  4. 实现 texture copy/resolve/mipmap，再补 query 与 indirect。
  5. 增加 GraphViz/JSON dump 与 GPU debug marker，直接可视化最终计划。
  6. 最后再做并行 command recording，因为它依赖稳定的资源与队列计划。
