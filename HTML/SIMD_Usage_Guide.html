0. 先看当前项目实际做了什么

SIMD 不是一种算法，而是同一条指令同时处理多个数据元素。当前 Math 库只把经过筛选的 float4x4 热点放进 SSE2 路径，公共类型仍保持普通、紧凑、可序列化的内存布局。

当前 ISA

x86/x64 SSE2，使用 128 位 __m128。

当前热路径

4x4 矩阵乘向量、乘矩阵与求逆。

当前回退

关闭 SIMD 或常量求值时执行标量 C++。

当前数据布局

行主序存储，列向量右乘：p' = M * p。

1. SIMD 的思维模型

把寄存器看作由若干 lane 组成的短数组。对两个寄存器做一次加法时，相同位置的 lane 同时相加。SSE 的 128 位寄存器能放 4 个 float，AVX2 的 256 位寄存器能放 8 个，AVX-512 能放 16 个。

A = [a₀, a₁, a₂, a₃] B = [b₀, b₁, b₂, b₃] A + B = [a₀+b₀, a₁+b₁, a₂+b₂, a₃+b₃]
寄存器与 lane 可视化

切换 ISA，观察单条 float 向量指令一次覆盖的数据量。

1.1 SIMD 适合什么

适合

大量元素执行同一种算术；矩阵、粒子、蒙皮、图像、音频、压缩和碰撞批处理。

不一定适合

数据量很小、分支极不一致、内存随机访问、算法主要受缓存或延迟限制。

1.2 向量化的三个层级

  1. 编译器自动向量化：保持普通循环，让编译器证明迭代互不依赖后生成向量指令。
  2. 编译器向量扩展或 std::simd：用接近 C++ 运算符的方式表达 lane 运算。
  3. intrinsic：直接调用与指令接近的函数，例如 _mm_add_ps。控制力最强，但架构耦合也最强。
SIMD 宽度不是理论加速倍数 4 个 lane 不等于必然快 4 倍。Load/Store、shuffle、水平归约、缓存未命中、前后依赖、调用开销和 CPU 执行端口都会消耗周期。

2.1 当前项目为何选择 SSE2

项目当前目标是 Windows/x64 学习型引擎。x64 ABI 保证 SSE2，因此不需要给 SSE2 路径写运行时 CPUID 分派；128 位宽度也与 float4、矩阵一行天然匹配。AVX2 更适合一次处理多个对象、多个顶点或多个粒子，而不只是把一个 float4 放进更宽寄存器。

2.2 不要按“更新”盲目选 ISA

  • 一个 4 分量向量用 AVX2 时，高 128 位可能空着，未获得双倍有效数据。
  • AVX-512 在部分微架构上可能触发更低频率，宽度收益需要实测。
  • 引擎发布版本通常选择一个广泛基线，再为热点提供 AVX2 等可选实现。
  • 跨 x86 与 Arm 时，应把算法接口和 ISA 实现分层，避免 intrinsic 泄漏到业务层。

3. Intrinsic 基础

intrinsic 是编译器提供的函数样式接口，通常映射到一条或少数几条机器指令。它不是普通库函数，不要根据函数名猜性能；最终应检查编译器生成的汇编。

3.1 x86 名称怎么读

_mm_add_ps │ │ └─ ps = packed single-precision，多个 float │ └───── add = 加法 └───────── _mm = 128 位 SSE 家族 _mm256_mul_ps → 256 位、8 个 float 的并行乘法 _mm_add_ss → scalar single，只更新最低 float lane
后缀含义常见类型例子
pspacked single多个 float_mm_mul_ps
pdpacked double多个 double_mm_add_pd
ssscalar single最低 float lane_mm_add_ss
epi32packed 32-bit integer有符号/按位整数 lane_mm_add_epi32
si128整个 128 位整数寄存器不强调 lane 类型_mm_loadu_si128

3.2 常见寄存器类型

#include <immintrin.h>

__m128  fourFloats;       // 128 bit: 4 × float
__m128d twoDoubles;       // 128 bit: 2 × double
__m128i integerBits;      // 128 bit: integer / raw bits
__m256  eightFloats;      // 256 bit: 8 × float, requires AVX
__m512  sixteenFloats;    // 512 bit: 16 × float, requires AVX-512F

// __m128i 本身不记录“这是 i8 还是 i32”。具体指令决定 lane 解释。

3.3 一条 intrinsic 不一定等于一条指令

_MM_TRANSPOSE4_PS 是宏，会展开为多条 unpack/move 指令；_mm_set_ps 也可能由常量加载、插入或 shuffle 构成。反过来，编译器也可能融合乘加、消除多余 Load，或把多个 intrinsic 重新安排。

4. Load、Store、对齐和数据布局

CPU 只能对寄存器执行大多数 SIMD 算术。数据先从内存 Load 到寄存器，完成运算后再 Store。算法是否快，常常取决于数据能否连续、批量地进入寄存器。

4.1 对齐与非对齐访问

const float* source = ...;
float* destination = ...;

__m128 a = _mm_loadu_ps(source);          // 任意合法 float 地址，读取 16 bytes
_mm_storeu_ps(destination, a);            // 任意合法 float 地址，写入 16 bytes

__m128 b = _mm_load_ps(alignedSource);    // 地址必须满足 16-byte alignment
_mm_store_ps(alignedDestination, b);       // 未满足对齐要求属于错误

现代 x86 上，未对齐访问若没有跨缓存行或页边界，通常与对齐访问接近；但“通常接近”不代表可以越界。_mm_loadu_ps 仍会读取完整 16 字节，即使你只需要 3 个 float。

float3 的典型越界问题 紧凑 float3 只有 12 字节。直接从最后一个独立 float3 执行 _mm_loadu_ps 会额外读取第 4 个 float，若对象后面没有可访问空间就是越界。当前矩阵路径加载的是完整 float4，不存在这个问题。

4.2 AoS 与 SoA

AoS：Array of Structures

Particle{x,y,z,life} 连续排列。读取一个对象全部字段方便，批量处理同一字段时需要 shuffle/gather。

SoA：Structure of Arrays

x[]、y[]、z[]、life[] 分开。每次 Load 得到多个对象的同一字段，适合批处理。

// AoS：每个对象的数据聚在一起。
struct ParticleAoS { float x, y, z, life; };
std::vector<ParticleAoS> particles;

// SoA：同一字段连续，可一次读取 4 或 8 个粒子的 x。
struct ParticleSoA {
    std::vector<float> x, y, z, life;
};

4.3 尾部处理

元素数量通常不是 SIMD 宽度的整数倍。SSE 每次处理 4 个 float，主循环走到 count / 4 * 4，余下 0 到 3 个元素用标量处理。也可用 padding 或 masked load，但必须保证不访问无效内存。

std::size_t i = 0;
for (; i + 4 <= count; i += 4) {
    // SIMD body
}
for (; i < count; ++i) {
    // Scalar tail
}

4.4 别名与内存生命周期

  • 不要通过不兼容的 C++ 指针类型解引用来“强转”寄存器，优先使用 Load/Store、std::bit_cast 或 memcpy。
  • 输入输出可能重叠时，编译器不能自由重排。明确 API 不重叠，或在支持的编译器上谨慎使用 restrict。
  • 对齐只描述地址边界，不代表对象后面存在足够的可读字节。

5. SIMD 的核心操作

5.1 构造、广播和算术

__m128 zero = _mm_setzero_ps();                    // [0, 0, 0, 0]
__m128 one  = _mm_set1_ps(1.0F);                 // [1, 1, 1, 1]
__m128 data = _mm_setr_ps(1, 2, 3, 4);         // 按阅读顺序构造

__m128 sum     = _mm_add_ps(data, one);            // [2, 3, 4, 5]
__m128 product = _mm_mul_ps(data, one);            // lane-wise multiply
__m128 minimum = _mm_min_ps(data, one);
__m128 root    = _mm_sqrt_ps(data);

5.2 Shuffle

shuffle 在 lane 之间重排数据。它是矩阵、复数、叉积等算法的核心，也常是吞吐瓶颈。_MM_SHUFFLE(z,y,x,w) 的参数阅读方向容易混乱，应先写测试确认输出 lane。

__m128 row = _mm_setr_ps(1, 2, 3, 4);
__m128 xxxx = _mm_shuffle_ps(row, row, _MM_SHUFFLE(0, 0, 0, 0));
// xxxx = [1, 1, 1, 1]：把 x 广播到全部 lane。

5.3 比较、mask 与 Select

SSE 比较结果不是 C++ bool4，而是每个 lane 全 1 或全 0 的位掩码。按位组合后可用 and/andnot/or 在不分支的情况下选择值。

__m128 mask = _mm_cmpgt_ps(value, threshold);
__m128 selected = _mm_or_ps(
    _mm_and_ps(mask, whenTrue),
    _mm_andnot_ps(mask, whenFalse));

// SSE4.1 可用 _mm_blendv_ps(whenFalse, whenTrue, mask)。
int laneBits = _mm_movemask_ps(mask);             // 提取 4 个符号位
bool any = laneBits != 0;
bool all = laneBits == 0b1111;

5.4 水平归约

lane-wise 运算保持 lane 独立；点积、总和、最小值需要把多个 lane 合成一个标量，这叫 reduction。归约会形成依赖链，不能简单按 lane 数估算收益。

// 当前项目 DotFloat3：只累加 x/y/z，忽略 w。
__m128 product = _mm_mul_ps(lhs, rhs);
__m128 sum = _mm_add_ss(
    product,
    _mm_shuffle_ps(product, product, _MM_SHUFFLE(1, 1, 1, 1)));
sum = _mm_add_ss(
    sum,
    _mm_shuffle_ps(product, product, _MM_SHUFFLE(2, 2, 2, 2)));
float dot = _mm_cvtss_f32(sum);

5.5 Gather、Scatter 和预取

  • Gather：根据多个索引读取离散地址。AVX2 提供 gather，但它通常明显慢于连续 Load。
  • Scatter：向多个离散地址写入；x86 到 AVX-512 才有更完整的硬件 scatter。
  • Prefetch：提示缓存提前读取。硬件预取器已很强，手动 prefetch 只有在有稳定距离且实测有效时才保留。

6. 当前 Matrix.hpp 的 SIMD 路径

当前矩阵采用行主序存储，数学约定是列向量右乘。矩阵乘矩阵的第 i 行可写成右矩阵四行的线性组合：

resultRowᵢ = lhsᵢ.x · rhsRow₀ + lhsᵢ.y · rhsRow₁ + lhsᵢ.z · rhsRow₂ + lhsᵢ.w · rhsRow₃
MultiplyMatrixRow 数据流

观察 LHS 行如何广播，并与 RHS 四行加权得到最终结果。

MATH_FORCE_INLINE __m128 MultiplyMatrixRow(
    __m128 lhsRow,
    __m128 rhsRow0,
    __m128 rhsRow1,
    __m128 rhsRow2,
    __m128 rhsRow3) noexcept {
    const __m128 xxxx = _mm_shuffle_ps(lhsRow, lhsRow, _MM_SHUFFLE(0, 0, 0, 0));
    const __m128 yyyy = _mm_shuffle_ps(lhsRow, lhsRow, _MM_SHUFFLE(1, 1, 1, 1));
    const __m128 zzzz = _mm_shuffle_ps(lhsRow, lhsRow, _MM_SHUFFLE(2, 2, 2, 2));
    const __m128 wwww = _mm_shuffle_ps(lhsRow, lhsRow, _MM_SHUFFLE(3, 3, 3, 3));

    __m128 result = _mm_mul_ps(xxxx, rhsRow0);
    result = _mm_add_ps(result, _mm_mul_ps(yyyy, rhsRow1));
    result = _mm_add_ps(result, _mm_mul_ps(zzzz, rhsRow2));
    return _mm_add_ps(result, _mm_mul_ps(wwww, rhsRow3));
}

6.1 一行里每一步发生了什么

  1. LoadFloat4(lhs.rows[i]) 把左矩阵一整行加载到 __m128。
  2. 四次 shuffle 分别得到 [x,x,x,x]、[y,y,y,y]、[z,z,z,z]、[w,w,w,w]。
  3. 每个广播寄存器与右矩阵对应行逐 lane 相乘，一条乘法同时算出四个输出列的贡献。
  4. 三次并行加法合并四份贡献，得到结果矩阵完整的一行。
  5. 右矩阵四行只加载一次，然后被左矩阵四行复用。

6.2 矩阵乘向量为何先转置

M * v 的结果是矩阵四列分别乘 v.x/y/z/w 后相加。内存里存的是四行，因此当前实现先用 _MM_TRANSPOSE4_PS 把四个行寄存器变成四个列寄存器，再做广播加权。另一种写法是每行做点积；两种数学等价，性能应由目标 CPU 实测。

6.3 constexpr 如何与 intrinsic 共存

MATH_FORCE_INLINE constexpr float4x4 operator*(
    const float4x4& lhs,
    const float4x4& rhs) noexcept {
#if MATH_MATRIX_HAS_SSE2
    if (!std::is_constant_evaluated()) {
        return detail::MultiplyFloat4x4(lhs, rhs);
    }
#endif
    return /* 展开的标量公式 */;
}

编译期常量表达式不能执行普通 x86 intrinsic，因此 std::is_constant_evaluated() 在常量求值时选择纯 C++ 路径；运行时选择 SSE2。这样保留一个公共 API，同时兼顾编译期计算与运行时速度。

6.4 4x4 求逆路径

TryInverseFloat4x4 把四行装入寄存器，以三维叉积和点积构造余子式与行列式。行列式接近 0 时返回 false；成功时只做一次标量除法，再把逆行列式广播到所有 lane。最后通过 _MM_TRANSPOSE4_PS 把中间列形式转回行主序输出。

接口不因 SIMD 改变 调用方仍使用 float4x4 和 TryInverse。intrinsic 被限制在 detail 内，这使单元测试、序列化和未来 NEON 实现都更容易维护。

7. 从标量代码到 SIMD 的完整示例

7.1 数组相加：最清楚的第一步

void AddArraysSSE(
    const float* lhs,
    const float* rhs,
    float* output,
    std::size_t count) noexcept {
    std::size_t i = 0;
    for (; i + 4 <= count; i += 4) {
        const __m128 a = _mm_loadu_ps(lhs + i);
        const __m128 b = _mm_loadu_ps(rhs + i);
        _mm_storeu_ps(output + i, _mm_add_ps(a, b));
    }

    // count 不是 4 的倍数时，处理剩余 0..3 个元素。
    for (; i < count; ++i) {
        output[i] = lhs[i] + rhs[i];
    }
}

这个循环没有跨迭代依赖，访问连续，SIMD 主体和标量尾部清晰。实际工程中应先查看编译器是否已经自动向量化；手写版本只有在性能、汇编或稳定性确实更好时才有保留价值。

7.2 一次计算 4 个点与平面的有符号距离

与其用一个寄存器表示一个点的 x/y/z/w，这里用四个寄存器表示 4 个点的 x、y、z 和常数 1，即 SoA。这样一次得到 4 个独立点的距离。

struct Plane { float nx, ny, nz, d; };

__m128 SignedDistance4(
    __m128 pointX,
    __m128 pointY,
    __m128 pointZ,
    const Plane& plane) noexcept {
    const __m128 nx = _mm_set1_ps(plane.nx);
    const __m128 ny = _mm_set1_ps(plane.ny);
    const __m128 nz = _mm_set1_ps(plane.nz);
    const __m128 d  = _mm_set1_ps(plane.d);

    __m128 result = _mm_mul_ps(pointX, nx);
    result = _mm_add_ps(result, _mm_mul_ps(pointY, ny));
    result = _mm_add_ps(result, _mm_mul_ps(pointZ, nz));
    return _mm_add_ps(result, d);
}

当结果与 0 比较时，可得到 4 个点位于平面哪一侧的 mask。将这种批处理扩展到 4 个 AABB、4 个包围球或 4 个视锥体平面，是引擎可见性检测中常见的 SIMD 设计。

7.3 点积：明确 w 是否参与

float Dot3SSE2(__m128 lhs, __m128 rhs) noexcept {
    const __m128 p = _mm_mul_ps(lhs, rhs);
    const __m128 y = _mm_shuffle_ps(p, p, _MM_SHUFFLE(1, 1, 1, 1));
    const __m128 z = _mm_shuffle_ps(p, p, _MM_SHUFFLE(2, 2, 2, 2));
    return _mm_cvtss_f32(_mm_add_ss(_mm_add_ss(p, y), z));
}
不要把 Dot3 和 Dot4 混为一谈 如果 w lane 未初始化，错误地把它加入点积会导致随机结果。API 应明确维度，并用单元测试覆盖 w 为非零、NaN 或填充值的情况。

7.4 分支改写为 mask 前先判断收益

// 将 value 限制到 [0, 1]，没有逐 lane 分支。
__m128 clamped = _mm_min_ps(_mm_max_ps(value, _mm_setzero_ps()), _mm_set1_ps(1.0F));

简单 min/max 很适合无分支表达；但复杂 if/else 若两边都计算再 blend，可能比可预测分支更慢。SIMD mask 解决的是 lane 分歧，不保证减少总工作量。

8. 运行时 CPUID 与多版本分派

SSE2 在 x64 可作为编译基线，但 AVX/AVX2/AVX-512 不能只看编译器宏。编译器宏说明“这个翻译单元允许生成什么指令”，CPUID 说明“当前 CPU 支持什么”，XGETBV 说明“操作系统会保存哪些扩展寄存器状态”。三者缺一不可。

8.1 AVX2 的完整判断顺序

  1. CPUID leaf 1：检查 OSXSAVE 和 AVX bit。
  2. 执行 XGETBV(0)：确认 XMM bit 1 与 YMM bit 2 都由 OS 启用。
  3. CPUID leaf 7 subleaf 0：检查 AVX2 bit。
  4. 若算法用 FMA，还要单独检查 CPUID leaf 1 的 FMA bit。
// MSVC/x86 示例：展示必要检查。生产代码还应封装最大 CPUID leaf 检查。
#include <intrin.h>

bool SupportsAvx2() noexcept {
    int registers[4]{};
    __cpuid(registers, 1);

    constexpr int OsxsaveBit = 1 << 27;
    constexpr int AvxBit     = 1 << 28;
    if ((registers[2] & (OsxsaveBit | AvxBit)) != (OsxsaveBit | AvxBit)) {
        return false;
    }

    const unsigned __int64 xcr0 = _xgetbv(0);
    if ((xcr0 & 0b110) != 0b110) {       // XMM + YMM state
        return false;
    }

    __cpuidex(registers, 7, 0);
    constexpr int Avx2Bit = 1 << 5;
    return (registers[1] & Avx2Bit) != 0;
}

8.2 推荐的引擎组织

using TransformPointsFunction = void (*)(
    const float4x4&,
    const float3*,
    float3*,
    std::size_t);

struct MathDispatch {
    TransformPointsFunction transformPoints = TransformPointsScalar;
};

MathDispatch BuildMathDispatch() {
    MathDispatch dispatch{};
    if (SupportsAvx2()) {
        dispatch.transformPoints = TransformPointsAvx2;
    } else if (SupportsSse2()) {
        dispatch.transformPoints = TransformPointsSse2;
    }
    return dispatch;
}
  • 启动时检测一次并保存函数指针，避免每个元素或每次函数调用都判断 CPUID。
  • 标量、SSE2、AVX2 实现使用相同输入输出契约，并运行同一套正确性测试。
  • MSVC 通常把 AVX2 实现放在单独的 .cpp，只对该文件启用 /arch:AVX2。
  • 不要让含 AVX2 指令的静态初始化或普通路径在分派前执行。
只检查 CPUID 的 AVX bit 仍可能崩溃 旧操作系统可能不保存 YMM 状态。必须同时检查 OSXSAVE 与 XGETBV。AVX-512 还需要确认 opmask、ZMM 高位等额外状态位。

9.1 当前项目的检测宏

#if !defined(MATH_DISABLE_SIMD) && \
    (defined(_M_X64) || (defined(_M_IX86_FP) && _M_IX86_FP >= 2) || defined(__SSE2__))

_M_X64 表示 MSVC x64，_M_IX86_FP >= 2 表示 32 位 MSVC 至少启用 SSE2，__SSE2__ 是 GCC/Clang 常见特性宏。MATH_DISABLE_SIMD 优先级最高，用来强制验证标量路径。

9.2 MATH_FORCE_INLINE 的边界

强制内联减少微小包装函数的调用开销，也让编译器看见更多常量和别名信息。但它不是性能保证；过度内联会增加代码体积、指令缓存压力和编译时间。只对非常小、非常热、封装 intrinsic 的函数使用。

9.3 Debug 与 Release 的区别

  • Debug 下寄存器值更容易观察，但未优化代码不能代表最终性能。
  • 性能测试必须使用 Release，并确认没有调试迭代器、ASan 或频繁日志。
  • 正确性测试应同时覆盖 SIMD 开启和关闭两种配置。

10. 如何正确测 SIMD 性能

当前仓库的 Benchmarks/MathBenchmarks.cpp 已包含预热、自动校准、7 次采样、中位数、最小值和结果消费，并与 DirectXMath 对照。这个框架比只计时一次可靠，但仍要理解测到的是什么。

10.1 当前可直接使用的命令

cmake -S . -B build_math -DMATH_BUILD_TESTS=ON -DMATH_BUILD_BENCHMARKS=ON
cmake --build build_math --config Release
ctest --test-dir build_math -C Release --output-on-failure

# Windows 多配置生成器：
.\build_math\Release\MathBenchmarks.exe

# 再构建纯标量版本，对比同一台机器、同一配置：
cmake -S . -B build_math_scalar -DMATH_ENABLE_SIMD=OFF -DMATH_BUILD_BENCHMARKS=ON
cmake --build build_math_scalar --config Release

10.2 吞吐、延迟和依赖链

延迟 latency

一条指令的结果要过多少周期才能被下一条依赖指令使用。连续 x = f(x) 主要测延迟。

吞吐 throughput

流水线稳定后每周期能启动多少条独立指令。多个独立累加器更接近吞吐测试。

矩阵反复写回自己会形成跨迭代依赖，适合衡量实际串行调用的延迟，却不能证明 CPU 能同时处理多少个独立矩阵。引擎应同时测试“单对象连续变换”和“批量独立对象”。

10.3 防止基准被编译器删除

  • 让结果流入外部可观察 sink，当前 benchmark 用 volatile 与 atomic_signal_fence。
  • 不要把输入全部写成可在编译期折叠的常量。
  • 查看生成汇编，确认循环和目标指令真的存在。
  • 记录 CPU 型号、编译器版本、参数和电源模式，避免跨环境直接比较数字。

10.4 FMA 与数值变化

a*b+c 可融合成一次 FMA，只进行一次舍入，通常更快且更精确，但结果最低位可能与分开的乘法加法不同。若 SIMD 和标量路径必须逐 bit 一致，应控制 contraction 策略；图形数学一般使用误差范围比较。

优化目标应是完整工作负载 一个 intrinsic 微基准快，不代表场景更新更快。最终应测真实对象数量、真实数据布局和真实调用链，并用 profiler 确认热点占比下降。

11.1 浮点环境：FTZ 与 DAZ

非常接近 0 的 subnormal 数能保持渐进下溢精度，但某些 CPU/路径处理较慢。MXCSR 的 FTZ（结果 flush to zero）和 DAZ（输入 denormals are zero）可改变行为。引擎若启用，应在工作线程入口统一设置并记录，不要让第三方库静默改变全局线程状态。

11.2 min/max 遇到 NaN

x86 MINPS/MAXPS 的 NaN 和相等值选择规则不完全等于 std::min/std::max 的直觉。若算法依赖 NaN 传播、正负零或精确 operand 选择，应查 Intel 指令文档并写针对性测试。

11.3 SSE 与 AVX 混用

旧式 SSE 指令写 XMM 时，与 AVX 写 YMM 后进入旧 SSE 代码可能产生 transition penalty。现代编译器通常在 AVX 函数边界插入 vzeroupper；手写汇编、跨编译选项对象文件和函数指针分派时仍需检查反汇编。

12. 一套可重复的 SIMD 优化流程

  1. 先用 profiler 找到真实热点，并记录它占整帧或整任务的百分比。
  2. 为标量实现准备单元测试、随机输入、边界值、NaN/Inf 和误差容限。
  3. 确认瓶颈是算术、分支还是内存；如果是缓存未命中，先处理数据布局。
  4. 检查 Release 编译器的自动向量化报告和汇编，避免重复手工优化。
  5. 选择最低必要 ISA；优先固定、清楚的基线，再添加可选高 ISA 版本。
  6. 设计向量化维度：一个对象的多个分量，还是多个对象的同一分量。
  7. 明确 Load/Store 范围、对齐契约、别名关系、尾部策略和 mask 语义。
  8. 先实现最简单正确的 intrinsic 版本，再减少 shuffle、Load 和依赖链。
  9. 让 SIMD 与标量实现执行同一套测试，并在多种 CPU 上运行。
  10. 用真实数据做 Release benchmark，分别观察延迟、吞吐与完整帧时间。
  11. 检查汇编、缓存 miss、分支、频率和执行端口，而不仅是源代码行数。
  12. 只有收益稳定、代码可维护时才保留；把 ISA 细节限制在内部实现层。

12.1 调试寄存器的实用方法

alignas(16) float lanes[4]{};
_mm_store_ps(lanes, value);
// 在调试器中观察 lanes[0..3]。只用于调试，不留在热循环。
  • 为 shuffle 单独写输入为 [1,2,3,4] 的测试，输出顺序一眼可见。
  • 用 AddressSanitizer 检查 Load/Store 越界，用 UBSan 检查普通 C++ 未定义行为。
  • MSVC Disassembly、Compiler Explorer 或 objdump/llvm-objdump 可确认最终指令。
  • Intel VTune、AMD uProf、Linux perf 可进一步查看缓存、分支和流水线指标。

12.2 对当前 Math 库的合理下一步

  • 保留 SSE2 作为 x64 基线，不必为单个 float4 盲目换成 AVX2。
  • 若蒙皮、粒子、剔除成为热点，设计一次处理 4/8 个对象的批量 API，再评估 SSE2/AVX2。
  • 增加随机矩阵对照测试，比较 SIMD、标量和高精度参考结果。
  • benchmark 增加多个独立矩阵累加器，分别报告依赖延迟与批量吞吐。
  • 需要 Arm 平台时，在内部增加 NEON 实现，保持公共 Vector/Matrix API 不变。
