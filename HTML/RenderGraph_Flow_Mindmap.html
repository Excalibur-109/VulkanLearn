RenderGraph 不是渲染器，而是帧任务编译器

上层声明资源如何被 pass 读写；编译器推导顺序、裁剪、生命周期和同步；后端只执行稳定计划。

4 种 Pass 3 类 Hazard 10 个编译阶段 3 个 RHI 后端
声明 → 编译 → 执行
声明 编译 计划 执行

边界：Graph 决定“先做什么、资源处于什么状态”；Workload 决定“具体画什么”；后端决定“翻译成哪种原生 API”。

核心职责

RenderGraph

它读取资源、Pass 和 Workload 声明，生成后端可直接线性消费的执行计划。

  • 不负责场景收集、材质分类或可见性判断。
  • 不在业务层暴露 Vulkan/D3D barrier。
  • 不把每帧变化的 draw 参数写入缓存计划。
源码：RenderGraph/RHIRenderGraph.cpp

为什么排序两次：先用显式边得到可信扫描顺序，再沿该顺序推导资源 Hazard，最后对完整依赖图排序。

编译入口

CompileRHIRenderGraph

编译成功后，后端不再进行拓扑排序，也不再通过字符串查找资源。

  • 任何阶段发现声明错误都会返回 errors。
  • 稳定排序让调试日志和测试结果可重复。
  • 输出只保留可跨帧缓存的静态结构。
源码：CompileRHIRenderGraph

运行时入口

RHIDevice::SubmitFrame

设备门面先处理结构缓存，再把“当前 Packet + 缓存 Plan”一起交给所选后端。

  • 拓扑不变时不重复编译。
  • 动态参数不会因为计划缓存而冻结。
  • 三个后端消费同一份抽象计划。

依赖、资源标志与 ExecutionPlan 速查

这些概念决定一个 Pass 为什么必须等待、为什么可以被裁剪，以及逻辑资源能否共享物理槽。

三类数据 Hazard

资源标志

Imported图外创建；Graph 不负责生命周期。
Exported结果要在图外继续使用，是裁剪根。
Transient仅在图内短期存在，可压缩生命周期。
AllowAliasing允许与不重叠且描述兼容的资源复用槽。
NeverCull即使输出无人读取也必须保留。

ExecutionPlan 的四类核心输出

当前实现边界：queue 字段已经进入计划，但 Vulkan、D3D11、D3D12 目前仍以一个主 command stream/list/context 执行整帧；真正异步 Compute/Transfer 的 queue batch 尚未实现。

依赖速查

RAW / WAR / WAW

Hazard 是同一资源上相邻读写之间的顺序约束，不等同于资源状态转换。

  • 依赖边解决“哪个 Pass 先执行”。
  • Transition 解决“访问何时可见、资源处于什么状态”。
  • 两者都由 Graph 声明自动推导。
