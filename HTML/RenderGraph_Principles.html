独立于 Vulkan、Direct3D、Metal 与任何 RHI

RenderGraph：把一帧意图编译成正确、高效的 GPU 工作计划

RenderGraph 是一个面向帧的声明式编译器。上层只陈述 Pass 读取或写入哪些逻辑资源；它推导依赖、删除无用工作、安排资源复用与同步，再让任意后端执行一份明确的计划。

阅读方向：左侧是“声明的世界”，右侧是“编译后执行的世界”。中心不是渲染 API，而是将二者连接起来的纯图算法。

1. RenderGraph 的本质

RenderGraph 不是一组画面效果，也不是图形 API 的薄封装。它是一次性的、面向单帧的依赖图编译：把“哪些工作产生或消费哪些资源”转化成可验证的顺序、内存与同步计划。

上层只需要声明

  • 本帧有哪些 Pass。
  • 每个 Pass 读、写或创建什么资源版本。
  • 访问用途：采样、颜色附件、深度写、拷贝等。
  • 哪些结果是可观察的最终输出。

编译器负责推导

  • 生产者和消费者之间的依赖边。
  • 对最终输出没有贡献、可以删除的 Pass。
  • 逻辑资源的生存区间和可复用物理内存。
  • 访问之间的状态转换、屏障与等待。
核心约束：Pass 不应隐式读取或写入图外状态。只有资源用途完整出现在声明中，编译器才有能力证明顺序正确、裁掉无效工作，并安全复用显存。

2. 图模型：Pass、逻辑资源和边

最常用的模型是二部图：一类节点是 Pass，另一类节点是逻辑资源版本。写入形成 Pass → ResourceVersion，读取形成 ResourceVersion → Pass。投影到 Pass 后，便得到一个有向无环图。

概念含义必须记录的信息不是
Pass一次有语义边界的 GPU 工作，例如绘制、计算或拷贝。队列偏好、读集合、写集合、命令回调。任意一段命令；粒度过细会增加调度成本。
逻辑资源图中引用的纹理、Buffer、加速结构等。描述、导入/临时属性、初始状态、子资源范围。固定的物理对象或唯一显存地址。
用途某 Pass 如何访问资源，例如采样读、附件写、拷贝源。访问类型、阶段、子资源、队列。单纯读写布尔值；用途决定同步细节。
边生产版本和消费 Pass 间的先后关系。资源版本、生产者、消费者、访问用途。人工维护的 Pass 排序数组。
pass.read(gbufferColor, Usage::Sampled);  pass.read(depthBuffer, Usage::Sampled);  hdr = pass.write(hdr, Usage::StorageWrite);

3. 资源版本：让依赖成为显式数据

资源版本是 RenderGraph 最重要的抽象。一个句柄不只指向“这张纹理”，还指向“这张纹理在第几次写之后的值”。读取者因此精确依赖某次生产，而不是模糊依赖某个资源名称。

导入 v0外部资源的已知初值。
Pass A 写 v1写入创建下一版本。
Pass B 写 v2若需要 A，显式读 v1。
Pass C 读 v2消费者连到精确版本。
导出 v2交给屏幕、读回或下一帧。

版本规则

read(handle) 保持版本；write(handle) 产生新版本；create() 产生无外部初值的首版本；import() 引入 v0。

依赖含义

RAW 是真实数据依赖：读者等待生产者。WAR 与 WAW 在显式版本模型中通常由版本链和物理复用约束体现；原地修改必须声明明确用途。

4. 图编译：从声明到计划的十个阶段

构图阶段追求易写和语义清楚；编译阶段追求正确性与可执行性。输出应尽量是稳定、紧凑的整数索引和批次数据，执行时不再做名称查找或依赖推导。

0. 规范化
冻结本帧声明，为资源、Pass 和访问记录分配紧凑索引；处理默认描述与子资源范围。
1. 校验
检查重复名称、非法描述、未初始化读取、不兼容用途、同一 Pass 内矛盾访问与循环引用。
2. 版本解析
为每次写生成版本，记录版本唯一的生产者，并把每次读绑定到指定版本。
3. 建依赖图
从“生产版本 → 消费 Pass”建边；必要时补充原地读写、队列和别名所要求的顺序边。
4. 标记输出根
确定最终可观察结果，例如屏幕、导出纹理、读回 Buffer 或下一帧保存的历史资源。
5. 反向裁剪
从输出根沿依赖边反向遍历，只保留能影响输出的 Pass 与版本。
6. 拓扑排序
对保留的 DAG 稳定排序；同样的输入得到同样的计划，便于调试和缓存。
7. 生存期分析
在排序后记录每个临时资源的 firstUse / lastUse，得到逻辑生存区间。
8. 物理分配
将不重叠且兼容的临时区间映射到同一物理堆、页或资源对象，生成别名边界。
9. 同步与批次
比较相邻访问的用途和队列，生成状态转换、内存屏障、所有权迁移、signal / wait 与提交批次。
10. 固化计划
输出 Pass 顺序、资源表、物理槽、每 Pass 前后同步和跨队列提交信息。
validate → resolveVersions → buildDependencies → cull → topologicalSort → lifetimes → allocateAndAlias → synchronization → CompiledPlan

5. 生命周期与内存别名：为什么它能省显存

逻辑资源不等于物理资源。只要两个临时资源的有效使用区间不重叠，并且大小、格式、采样数、对齐和内存类型兼容，它们就可以复用同一块物理内存。

逻辑生存期

从资源版本第一次被使用开始，到最后一次被使用结束。创建得早，不代表必须占用到帧尾。

物理槽

分配器维护可复用槽。新资源优先放入已结束且描述兼容的槽；否则才扩容。

别名屏障

同一物理区域从 A 切换给 B 时，某些平台需要显式别名屏障，防止缓存或元数据错误继承。

安全条件：“逻辑名字不同”不是别名条件；必须证明使用区间不重叠，并满足 API 的内存兼容要求。导入资源、历史资源和被外部持有的资源通常不能按帧随意别名。

6. 同步：从用途变化推导 Barrier

屏障不是资源的固有属性，而是两次访问之间的关系。编译器为连续访问记录 stage、access、layout/state、queue 与子资源范围，再在边界处生成最小同步。

前一次用途后一次用途必须解决的问题典型同步内容
颜色附件写入片元采样读取写入结果必须对纹理读取可见。写入阶段 → 采样阶段；颜色写 → 着色器读；布局/状态转换。
计算存储写入计算存储读取同一队列内的 storage 数据依赖。内存可见性屏障，通常无需布局转换。
拷贝写入深度附件写入传输完成后才能作为深度目标使用。传输访问 → 深度测试访问；目标状态转换。
队列 A 写入队列 B 读取顺序、可见性和资源所有权都要跨队列建立。A signal，B wait；必要时 release / acquire。

同步粒度应尽可能使用真实子资源范围，例如 mip、array layer 或 Buffer range。把整张图像一律当作单一状态虽然简单，却会制造不必要屏障，阻断本可并行的工作。

7. 多队列调度：并行不是免费获得的

当硬件提供图形、计算和传输队列时，RenderGraph 可以把没有依赖的工作放进不同队列重叠执行。但跨队列需要额外同步，且共享内存带宽可能成为瓶颈，因此调度器必须比较收益和成本。

适合异步的工作

长时间后处理、粒子模拟、纹理流送、预计算，以及与图形主链只有少量边界依赖的计算任务。它们应尽早开始、尽晚等待。

常见反效果

极短计算 Pass、频繁读写同一资源、过多跨队列往返，以及为了并行而拆得太碎的 Pass，会让同步和缓存抖动吞掉收益。

Graphics: GBuffer ── wait ComputeLighting ─→ ToneMap  |  Compute: SSAO ─→ ComputeLighting  |  只在真实依赖边上 signal / wait。

8. 编译与执行必须分离

RenderGraph 的核心价值在于把昂贵的全局分析放在编译期，把执行期变成对计划的线性解释。执行器应知道“做什么”，但不应在命令录制时重新决定“为什么先做这个”。

Compiled Plan 包含

排序后的 Pass 索引、逻辑到物理资源映射、资源创建与销毁时机、Pass 前后同步、提交批次及跨队列等待关系。

每帧动态数据包含

相机常量、draw 列表、上传字节、外部资源当前句柄、屏幕尺寸和 Pass 回调参数。这些不应被错误缓存到结构计划中。

缓存的正确边界

资源描述、Pass 拓扑和访问用途不变时，图结构通常可按 hash 缓存。分辨率、格式、启用效果、队列偏好或资源用途变化时必须失效并重新编译。动态常量变化不必重编译，但执行时必须读取本帧数据。

9. 一个完全通用的帧例子

下例不依赖任何 RHI 或图形 API。它只说明资源版本和依赖；任何后端都可把编译结果翻译成自己的 command buffer、encoder 或 command list。

导入屏幕 v0外部可呈现目标。
GBuffer写 Albedo v1、Normal v1、Depth v1。
Lighting读三个 v1，写 HDR v1。
ToneMap读 HDR v1，写屏幕 v1。
导出屏幕 v1作为本帧输出交给呈现系统。

若另有调试 Pass 写入临时纹理但没有被任何导出结果读取，反向裁剪会删除它。若 SSAO 只读 Depth v1、写 AO v1，而 Lighting 读 AO v1，则编译器把 SSAO 放在 Lighting 之前；若硬件和同步成本允许，SSAO 可以和不依赖它的图形工作并行。

10. 设计边界、调试与测试

RenderGraph 很适合解决帧内资源依赖，但它不能替代所有渲染架构。明确边界能避免把它变成难以维护的全局状态机。

它应该管理

帧内 Pass、瞬态资源、导入资源访问、同步计划、跨队列批次与可视化调试信息。

它不应猜测

游戏对象生命周期、业务逻辑、材质系统的全部语义、隐式全局状态和跨帧历史资源创建策略。

优先测试

依赖方向、裁剪结果、拓扑稳定性、未初始化读取、别名合法性、Barrier 位置、跨队列 wait/signal 和缓存失效。

可视化是必要能力：调试视图至少应展示 Pass DAG、资源版本链、裁剪后的 Pass、每个资源的生存区间、物理槽复用、Barrier 和队列批次。能看见编译结果，RenderGraph 才容易被信任和维护。
RenderGraph 原理文档：抽象层只使用 Pass、资源版本、用途和编译计划，不依赖任何特定 RHI 或图形 API 的类型名。
