GLSL Shader 写法指南

这份指南面向 Vulkan 风格 GLSL 450，重点讲 shader 文件结构、关键字、资源绑定、顶点/片元/计算 shader 的组织方式，以及如何把公共代码拆成 include 文件。

常用关键字

关键字用途示例
#version 450指定 GLSL 版本。Vulkan 常用 450 起步。#version 450
layout(location = n)声明顶点输入、阶段输出和片元输出位置。layout(location = 0) in vec3 inPosition;
layout(set = x, binding = y)声明 Vulkan descriptor set 和 binding。layout(set = 0, binding = 0) uniform SceneUniforms
uniform只读常量数据，通常对应 uniform buffer 或 sampler。uniform sampler2D baseColorMap;
bufferstorage buffer，可保存数组、实例数据、材质数据。readonly buffer ObjectBuffer { ObjectData objects[]; };
readonly/writeonly约束 storage buffer/image 的访问方向，便于编译器优化和验证。readonly buffer
flat禁用插值，适合材质索引、对象 ID。flat out uint outMaterialIndex;
gl_Position顶点 shader 必须写出的裁剪空间位置。gl_Position = scene.viewProjection * worldPosition;
gl_InstanceIndex实例化绘制时当前 instance 下标。objectData.objects[gl_InstanceIndex]
gl_GlobalInvocationIDcompute shader 当前全局线程 id。uvec3 id = gl_GlobalInvocationID;

语义对应

GLSL 没有 HLSL 那种写在变量后面的 : SV_Position、: TEXCOORD0 语义。GLSL 把“系统语义”做成内建变量，把“自定义阶段传递语义”做成 layout(location = n)。所以学习 GLSL 时要记的是“内建变量 + location”，不是 HLSL 的语义后缀。

HLSL 语义GLSL 写法用途
SV_Positiongl_Position顶点 shader 输出裁剪空间位置。
SV_Target0layout(location = 0) out vec4 outColor片元 shader 输出到第 0 个 color attachment。
SV_VertexIDgl_VertexIndex当前顶点编号，常用于全屏三角形。
SV_InstanceIDgl_InstanceIndex当前实例编号，常用于读取对象数组。
SV_DispatchThreadIDgl_GlobalInvocationIDcompute shader 当前全局线程 id。
SV_GroupThreadIDgl_LocalInvocationIDcompute shader 在线程组内部的局部线程 id。
SV_GroupIDgl_WorkGroupIDcompute shader 当前线程组 id。
POSITIONlayout(location = 0) in vec3 inPosition顶点输入位置。语义名由 HLSL 使用，GLSL 只看 location。

TEXCOORD0layout(location = 0) out/in vec2 uv普通阶段间插值变量。HLSL 用语义名，GLSL 用 location 配对。

struct VSOutput {
    float4 position : SV_Position;
    float2 uv : TEXCOORD0;

    outUv = inUv;

阶段结构

顶点 shader

顶点 shader 做三件事：读取 vertex attribute，读取对象变换，把结果输出给后续阶段。不要在顶点 shader 中写复杂光照，除非目标是低成本或特殊风格。

layout(location = 0) in vec3 inPosition;
layout(location = 0) out vec3 outWorldPosition;

void main() {
    vec4 worldPosition = object.localToWorld * vec4(inPosition, 1.0);
    outWorldPosition = worldPosition.xyz;
    gl_Position = scene.viewProjection * worldPosition;
}

片元 shader

片元 shader 负责材质读取、光照、阴影、透明度和最终颜色输出。PBR 代码建议放在公共 include 中，主函数只保留数据组装和结果输出。

计算 shader

compute shader 使用 layout(local_size_x = ...) 声明线程组大小。CPU dispatch 的 group 数量乘以 local size，得到 GPU 实际线程数量。

资源绑定

建议把 set 固定成清晰的层级：set = 0 放每帧/每视图数据，set = 1 放对象和材质数组，set = 2 放贴图和采样器。

Set建议内容原因
0SceneUniforms、LightUniforms、shadow map每帧或每个 camera/pass 更新一次。
1ObjectBuffer、MaterialBuffer绘制大量对象时使用 instance id 或 material id 索引。
2纹理数组、sampler、IBL 贴图材质贴图通常多、变化频率低，适合独立管理。

代码整理规则

  • 统一命名：输入用 inPosition，输出用 outWorldPosition，uniform 实例用 sceneData。
  • 公共函数按职责拆分：数学和颜色放 common.glsl，PBR 放 pbr_lighting.glsl，数据布局放 scene_layout.glsl。
  • 所有绑定点写在文件顶部，避免资源藏在函数中间。
  • 矩阵约定要固定：示例按列主序矩阵和 projection * view * model * position 的顺序写。
  • 材质和对象结构要与 C++ 侧字节布局严格匹配，必要时用 vec4 对齐。
  • 不要把 gamma 校正、tone mapping 混进 PBR 光照函数，后处理阶段单独处理更清楚。

检查清单

  1. 每个 shader 是否声明了正确的 #version。
  2. 顶点输入 location 是否和 C++ 的 vertex layout 完全一致。
  3. set/binding 是否和 RenderDefinitions 中的 BindGroupLayout 对齐。
  4. uniform/storage buffer 结构是否满足 std140/std430 对齐。
  5. 深度纹理采样是否使用 sampler2DShadow 或手动比较，两者不要混用。
  6. 颜色贴图是否在线性空间参与光照，只在最终输出前做 gamma/tone mapping。
