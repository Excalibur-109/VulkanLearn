Vulkan / D3D11 / 自研微型游戏引擎学习路线

目标不是只会调 API，而是从 0 用 C++ 写出一个有编辑器 GUI、场景、资源、渲染、运行时的微型游戏引擎。 当前项目已经有统一渲染定义、Vulkan 后端、D3D11 后端和 PBR 阴影示例，可以直接作为长期学习主线。

学习原则 基于当前代码 Vulkan 核心 D3D11 核心 渲染器演进 引擎架构 自研 GUI 编辑器能力 资源系统 调试方法 阶段性产物

一、学习原则

你要掌握 Vulkan、D3D11 和游戏引擎，最有效的方式不是孤立背 API，而是围绕“引擎必须解决的问题”学习。 每个功能都同时问四个问题：数据怎么组织，GPU 资源怎么创建，shader 怎么读取，渲染状态怎么保证正确。

核心判断标准：看到一个渲染错误时，你能系统地区分它是 CPU 数据错、GPU 资源错、绑定错、shader 错、状态错、同步错，还是坐标系错。
  • Vulkan 学的是显式资源、显式同步、显式状态转换。
  • D3D11 学的是状态机、绑定槽、资源视图、debug layer 反馈。
  • 引擎学的是抽象边界、资源生命周期、编辑器数据流和运行时数据流。
  • GUI 学的是输入、布局、绘制、文本、交互状态和可维护控件系统。

二、基于当前项目怎么学

当前项目最有价值的结构是：

文件 学习重点
RenderDefinitions.hpp 把它当成自研引擎的渲染 API 规范，理解每个 desc、handle、enum、frame packet 的职责。
RenderVulkan.cpp 学习 Vulkan 后端如何把统一抽象落到 VkBuffer、VkImage、descriptor、pipeline、dynamic rendering、barrier。
RenderD3D11.cpp 学习 D3D11 后端如何把统一抽象落到 buffer、texture view、input layout、shader stage、OM 状态。
PbrShadowExample.cpp 学习 Vulkan 端完整一帧：资源创建、上传、shadow pass、lighting pass、present。
PbrShadowD3D11Example.cpp 学习 D3D11 端完整一帧，对照 Vulkan 看同一个渲染任务在不同 API 下的差异。
Shaders/* 学习 CPU/GPU 数据布局、HLSL/GLSL 坐标差异、PBR、shadow map、资源绑定。

这次修复里出现的 std430 stride、glm::uvec4 对齐、Vulkan 深度范围、Y 翻转、D3D11 SRV offset， 都是后面写引擎时会反复遇到的真实问题，建议单独整理成“图形 bug 案例库”。

三、Vulkan 必须掌握的内容

基础对象

  • VkInstance、validation layer、extension。
  • VkPhysicalDevice、queue family、feature、limit。
  • VkDevice、graphics queue、present queue、transfer queue。
  • VkSurfaceKHR、VkSwapchainKHR、swapchain image。

资源模型

  • VkBuffer、VkImage、VkImageView、VkSampler。
  • device local、host visible、staging upload。
  • image layout、access mask、pipeline stage。
  • buffer/image copy、mipmap、MSAA resolve。

Descriptor 和 Pipeline

  • 理解 DescriptorSetLayout 是 shader 资源接口，PipelineLayout 是 pipeline 能看到的 set 组合。
  • 理解 uniform buffer、storage buffer、combined image sampler、storage image 的差异。
  • 掌握 vertex input、input assembly、raster、depth stencil、blend、dynamic state。
  • 能解释为什么同一个 shader layout 变了，pipeline layout、descriptor set 和绑定代码都要一起变。

同步和帧调度

  • 先掌握 fence、binary semaphore、timeline semaphore 的用途。
  • 再掌握 barrier：old layout、new layout、src/dst stage、src/dst access。
  • 理解 frames in flight：每帧独立 command buffer、uniform buffer、descriptor、fence。
  • 能用 RenderDoc 看清某张 texture 在每个 pass 的 layout 和内容。

四、D3D11 必须掌握的内容

D3D11 的难点不是同步，而是“当前 context 状态到底完整不完整”。它是状态机，draw call 是否有效取决于 IA、shader、resource、sampler、RS、OM 全部状态。

  • Device、ImmediateContext、SwapChain、BackBuffer、RTV、DSV。
  • InputLayout 必须和 HLSL vertex shader 输入签名匹配。
  • ConstantBuffer 必须 16 字节对齐，结构体字段也要按 HLSL packing 理解。
  • SRV/RTV/DSV/UAV 是同一资源的不同视图，绑定冲突时需要解绑。
  • SamplerComparisonState 和 shadow map 采样规则要和 depth format、compare op 对齐。
  • Debug layer 的 warning 要逐条读，不要忽略。
建议写一个 D3D11 状态检查清单：每次 draw 前确认 input layout、topology、vertex/index buffer、VS/PS、constant buffer、SRV、sampler、RTV/DSV、viewport、scissor、blend/depth/raster state。

五、渲染器演进路线

  1. 基础 清屏、三角形、indexed mesh、texture quad。
  2. 材质 base color、normal map、metallic roughness、AO、emissive。
  3. 光照 directional light、point light、spot light、shadow map。
  4. PBR GGX、Fresnel、IBL、prefilter environment、BRDF LUT。
  5. 调试 depth view、normal view、material id、wireframe、bounding box。
  6. 性能 material sort、pipeline sort、instance draw、dynamic buffer ring。
  7. 后续 deferred rendering、clustered lighting、GPU culling、post process。

不建议一开始就追 deferred、bindless、GPU driven。先让一个 forward renderer 稳定、清晰、可调试，再叠复杂度。

六、自研微型引擎架构

模块 职责 建议先做到什么程度
Core 日志、断言、基础类型、句柄池、文件路径、序列化。 够稳定，不追复杂内存分配器。
Platform 窗口、输入、时间、剪贴板、文件对话框。 先 Windows + GLFW 或 Win32，后面再抽象。
Render 统一渲染 API、Vulkan/D3D11 backend、RenderGraph。 沿用当前项目继续完善。
Scene Entity、Transform、Camera、Light、MeshRenderer。 先做简单 ECS 或对象组件模型，不要过度设计。
Asset mesh、texture、shader、material、scene 文件加载和缓存。 先支持自定义 JSON + 外部图片/模型。
Editor Hierarchy、Inspector、SceneView、GameView、AssetBrowser、Console。 先能编辑 transform/material 并保存场景。
Runtime 游戏循环、组件更新、场景加载、输入分发。 先能 Play/Stop 一个简单场景。

七、自研 GUI 路线

如果 UI 也从 0 实现，建议先做 immediate mode GUI。它更容易和编辑器工具结合，调试成本低，适合自研引擎早期。

GUI 最小内核

  • 输入状态：mouse position、button down/up、wheel、keyboard、text input。
  • 交互状态：hot item、active item、focused item、keyboard focus。
  • 绘制命令：rect、line、text、image、clip rect、scissor。
  • 布局：row、column、same line、padding、spacing、scroll view。
  • 控件：label、button、checkbox、slider、input text、combo、tree、table。

文本系统

  • 先支持 UTF-8 字符串、字体 atlas、glyph cache。
  • 再做中文显示、fallback font、换行、裁剪、选区、光标。
  • 文本输入要处理 IME，这对中文编辑器非常重要，可以后置但不能永远忽略。

编辑器专用控件

  • Docking、splitter、menu bar、toolbar、context menu。
  • Hierarchy tree、Inspector property grid、Asset grid、Console list。
  • Drag drop、rename、multi-select、search/filter。

八、Unity 风格编辑器最小闭环

  1. 窗口分区：SceneView、GameView、Hierarchy、Inspector、AssetBrowser、Console。
  2. Hierarchy 显示场景中的 entity，支持创建、删除、改名、父子关系。
  3. Inspector 显示 Transform、Camera、Light、MeshRenderer、Material。
  4. SceneView 支持相机飞行、选择物体、显示 bounding box。
  5. Gizmo 支持 translate，后续再做 rotate/scale。
  6. AssetBrowser 能浏览资源目录，拖 mesh/material 到场景对象。
  7. 场景能保存到文件，重启编辑器后恢复。
  8. Play/Stop 模式：编辑器数据复制到 runtime world，停止后不污染编辑态。

九、资源系统建议

  • 资源 ID 不要直接用路径，建议使用稳定 GUID。
  • 源文件和导入产物分开：例如 Assets/ 存源资源，Library/ 存缓存。
  • 材质保存 shader、参数、texture 引用。
  • mesh 保存 vertex/index/submesh/bounds。
  • texture 保存格式、尺寸、mipmap、sRGB 标志。
  • scene 保存 entity、component、asset reference。

早期可以用 JSON，后期再换二进制。重点是先把依赖关系和 reload 流程设计清楚。

十、调试和验证方法

  • 每个 GPU 对象都设置 debug name。
  • Vulkan 永远开 validation layer 做开发。
  • D3D11 开 debug layer，并学会读 warning。
  • 每个 pass 都能单独显示输出：shadow map、depth、normal、base color。
  • 每个 draw 都能定位：mesh、material、pipeline、bind group、shader。
  • 遇到黑屏先改 clear color，再改 shader 输出常量色，再查 vertex shader 位置。
  • 遇到物体消失查：索引范围、矩阵、深度、剔除、viewport/scissor、绑定资源。
最推荐的固定流程：CPU 数据打印一份，RenderDoc 抓一帧，检查 draw call 输入，检查 shader resource，检查 VS 输出，检查 PS 输出，检查 OM/depth/blend。

十一、建议做出的阶段性产物

  1. 渲染映射表：列出每个 RenderDefinitions 抽象在 Vulkan 和 D3D11 中分别落到什么对象。
  2. GPU 调试笔记：记录每次黑屏、错位、阴影错误的根因和排查路径。
  3. Mesh Viewer：能加载模型，切换材质，显示 normal/wireframe/bounds。
  4. Material Inspector：用自研 GUI 改 PBR 参数并实时更新 GPU buffer。
  5. Scene Editor 原型：Hierarchy + SceneView + Inspector + Save/Load。
  6. Runtime Demo：编辑器里摆场景，点击 Play，运行一个简单游戏交互。

只要这些产物逐个完成，你会自然覆盖 Vulkan、D3D11、GUI、资源系统、场景系统和编辑器架构。 不需要先把所有理论学完再动手，应该每学一块就做一个能跑、能调试、能保存的最小功能。
